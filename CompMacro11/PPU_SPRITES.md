# Работа с ПП (периферийным процессором) для аппаратных спрайтов в 8 цветах

Документ собран по исходникам эмулятора UKNCBTL (Memory.cpp, Board.cpp) и
«Ресурсы УКНЦ Глава 1». Цель: выводить цветные спрайты через аппаратный
спрайтовый механизм ПП, недоступный с ЦП.

## 1. Почему именно ПП

Аппаратный спрайтовый механизм — это четыре регистра в **странице ввода-вывода ПП**:
их НЕТ у ЦП. Значит код, использующий их, должен исполняться на ПП.

| Регистр | Адрес (ПП) | Назначение |
|---------|-----------|------------|
| РО — октет точек | 177024 | «срез» спрайта: 8 бит = 8 точек по горизонтали |
| РТ — код цвета точки | 177016 | цвет спрайта, 3 бита (план 0,1,2) → значение 0..7 |
| РФ — код цвета фона | 177020 / 177022 | T-слово фона (8 точек × 3 плана) |
| маска планов | 177026 | запрет записи в план (бит0=план0…) |

Регистр адреса/данных ПП для доступа к видеопланам:
- РА = 177010 (адрес в ВОЗУ)
- РД план0 = 177012, планы1&2 = 177014

## 2. Точная семантика (из эмулятора Memory.cpp)

### Фаза ЧТЕНИЕ фона — чтение 177024 (TST @#177024)
При чтении РО эмулятор берёт 3 байта планов по адресу РА (177010) и
раскладывает в РФ (177020/177022) по триадам: каждой из 8 точек — 3 бита
(план0=бит0, план1=бит1, план2=бит2). РО при чтении даёт 0.
**Перед чтением загрузить РА нужным адресом.**

### Фаза МОДИФИКАЦИЯ-ЗАПИСЬ — запись байта в 177024 (MOVB b,@#177024)
Эмулятор делает ровно это (расшифровка кода):
```
planebyte[0..2] = текущий фон из РФ (разложенный обратно в 3 байта планов)
// наложить спрайт: там где бит РО=1, поставить цвет РТ
planebyte[0] &= ~РО;  if (РТ & 1) planebyte[0] |= РО;
planebyte[1] &= ~РО;  if (РТ & 2) planebyte[1] |= РО;
planebyte[2] &= ~РО;  if (РТ & 4) planebyte[2] |= РО;
// записать в планы (если не запрещено маской 177026)
if (!(маска&1)) план0[РА] = planebyte[0];
if (!(маска&2)) план1[РА] = planebyte[1];
if (!(маска&4)) план2[РА] = planebyte[2];
```
Итог: в октете по адресу РА точки, где РО=1, перекрашиваются в цвет РТ (0..7),
остальные точки фона не трогаются. Цвет — полные 3 бита → **8 цветов**.

## 3. Алгоритм вывода одного спрайта (8×N пикселей)

Спрайт = N байт (по строкам), каждый байт — октет точек одной строки.
Для одноцветного спрайта цвета C:
```
РТ (177016) = C            ; цвет спрайта 0..7, один раз
для каждой строки спрайта y:
    РА (177010) = адрес_ВОЗУ(x, y)   ; позиция в видеопамяти
    TST @#177024                      ; ФАЗА ЧТЕНИЕ: загрузить фон в РФ
    MOVB байт_строки, @#177024        ; ФАЗА ЗАПИСЬ: наложить спрайт
```
Для многоцветного — менять РТ перед каждой записью (фон читать раз на октет).

Стирание спрайта = записать назад сохранённый фон (или РТ=цвет_фона).

## 4. Как считать адрес в ВОЗУ

Из таблицы строк: строке экрана соответствует базовый адрес в ВОЗУ.
Один октет (8 точек) = 1 адрес. По горизонтали +1 адрес = +8 точек.
Адреса ≥100000 — планы (верхнее ОЗУ), <100000 — нижнее ОЗУ ЦП+ПП.
Формат экрана (из УО): по умолчанию 80 октетов в строке (640 точек).

## 5. Схема ЦП ↔ ПП (наша архитектура)

ЦП НЕ видит ОЗУ ПП и регистры ПП напрямую. Связь — через каналы (Глава 3):
- К0 (177566 ЦП → 177060 ПП) — занят системным терминалом
- **К1 (176666 ЦП → 177062 ПП) — свободен, его берём для команд спрайтов**

### Модель работы (твоя схема)
1. **Загрузка кода на ПП.** Код спрайт-движка должен оказаться в ОЗУ ПП
   (адреса 0..77777, ~22КБ свободно). Способы: через монитор ПП, либо ПП
   сам копирует из ВОЗУ. На эмуляторе — проще предзагрузить.
2. **Обработчик прерывания К1 на ПП.** ПП ставит в вектор приёмника К1
   (ячейка 330) адрес своего обработчика. При посылке от ЦП срабатывает
   прерывание — ПП читает байт-команду и рисует спрайт.
3. **Вызов с ЦП.** ЦП пишет команду в 176666 (источник К1):
   например байт = номер спрайта или пакет (id, x, y). ПП ловит, рисует.

### Протокол команд (проект)
Простой вариант — слать пакет по К1:
```
байт1 = команда (1=draw, 2=erase, 3=move)
байт2 = id спрайта (какой битмап из предзагруженных)
байт3 = x (в октетах 0..79)
байт4 = y (строка)
```
ПП-обработчик собирает пакет и вызывает свою draw-процедуру.

## 6. Предзагруженные спрайты (твой вариант «простые вызовы»)

Лучший путь для скорости: битмапы спрайтов лежат в ОЗУ ПП заранее
(таблица: id → адрес битмапа + размер). Тогда вызов с ЦП — это просто
«нарисуй спрайт id в (x,y)», 3-4 байта по каналу. Вся тяжёлая работа
(чтение фона, наложение, запись планов) — на ПП, параллельно с ЦП.

Для Battle City: танки (4 направления), пуля, стены, орёл — набор битмапов
в ОЗУ ПП, ЦП шлёт только «двигай танк id в x,y».

## 7. Что проверить на железе (план)

1. Минимальный тест: ПП рисует ОДИН цветной октет в (x,y) фиксированного цвета.
   Проверить, что 8 цветов реально выводятся.
2. Спрайт из нескольких строк (танк 8×8 или 16×16).
3. Канал К1: ЦП шлёт байт → ПП рисует. Синхронизация.
4. Движение: стереть (восстановить фон) + нарисовать в новой позиции.
