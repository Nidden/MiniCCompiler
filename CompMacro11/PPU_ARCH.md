# Архитектура ЦП↔ПП для аппаратных спрайтов (Battle City)

Подтверждено исходниками эмулятора (Board.h: `m_pRAM[3]` — три плана по 64КБ,
ОБЩИЕ для обоих процессоров) и документацией. Схема пользователя рабочая.

## Ключевая идея (решение пользователя)

ЦП не видит ОЗУ ПП напрямую. НО оба процессора видят **видеопланы** (общая
физическая память `m_pRAM`):
- ЦП пишет в планы 1,2 через РА/РД: 176640 (адрес), 176642 (данные пл.1,2)
- ПП читает планы 0,1,2 через РА/РД: 177010 (адрес), 177012 (пл.0), 177014 (пл.1,2)

→ **Передача кода и графики ЦП→ПП идёт через планы.** ЦП кладёт байты в план,
ПП их оттуда забирает. Канал нужен только для коротких команд-сигналов.

## Три задачи (по уточнению пользователя)

### 1. Передать и запустить управляющую программу на ПП
- ЦП пишет машинный код ПП-движка в выделенную область плана (через РА/РД).
- ПП копирует этот код из плана в своё ОЗУ (адреса 0..77777, ~22КБ свободно).
- ПП передаёт управление на скопированный код (JMP).
- Тонкость: нужен крошечный начальный загрузчик на ПП, который умеет
  «прочитать код из плана и запустить». Где он берётся — см. открытые вопросы.

### 2. Канал для команд ПП
- К0 занят терминалом. **Берём К1** (двунаправленный, свободен).
  - ЦП источник: состояние 176664, данные 176666
  - ПП приёмник: данные 177062, вектор прерывания 330
- ЦП шлёт короткую команду (1 байт-сигнал): «есть новый кадр/пакет в плане».
- ПП по прерыванию К1 читает сигнал, идёт в план за деталями (что рисовать).

### 3. Графику и код передаём через планы
- Битмапы спрайтов (танки, пули, стены, орёл) ЦП кладёт в план заранее.
- ПП-движок держит таблицу: id спрайта → адрес битмапа в плане/ОЗУ ПП.
- На кадр ЦП обновляет в плане «список объектов» (id, x, y), шлёт сигнал по К1.

## Поток данных (один кадр)

```
ЦП (игровая логика):
  1. посчитал позиции танков/пуль
  2. записал в план таблицу объектов: [id,x,y, id,x,y, ...]  (через РА/РД 176640/176642)
  3. послал байт-сигнал по К1 (MOV сигнал, @#176666)

ПП (спрайт-движок, по прерыванию К1):
  1. прочитал сигнал из 177062
  2. читает таблицу объектов из плана (РА/РД 177010/177012/177014)
  3. для каждого объекта: стереть старую позицию, нарисовать в новой
     (спрайтовый механизм 177016/177020/177024 — см. PPU_SPRITES.md)
  4. RTI — вернулся, ждёт следующий сигнал
```

ЦП и ПП работают параллельно: пока ПП рисует, ЦП считает следующий кадр.

## Карта памяти под задачу (черновик)

| Область | Где | Что |
|---------|-----|-----|
| Видеопланы 0,1,2 | верхнее ОЗУ (РА≥100000) | сам экран (картинка) |
| Буфер обмена | план 1 или 2, своя зона | таблица объектов, код для ПП |
| ОЗУ ПП | 0..77777 | ПП-движок + битмапы спрайтов |
| Битмапы | ОЗУ ПП | танк×4, пуля, кирпич, сталь, орёл |

## Спрайты Battle City (8 цветов)

Цвет точки РТ (177016) = 0..7 → 8 цветов. Для танков:
- игрок — жёлтый, враги — серый/зелёный/красный (разные типы)
- кирпич — красный, сталь — серая, орёл — белый/жёлтый
Многоцветный спрайт: менять РТ между фазами записи (см. PPU_SPRITES.md §3).

## Открытые вопросы (нужны факты)

1. **Начальный запуск кода на ПП.** Как на реальной машине / в UKNCBTL
   заставить ПП исполнять наш код? Варианты:
   - через монитор ПП (ручной ввод загрузчика)
   - через системный механизм загрузки
   - спец-вектор/прерывание
   Это главный нерешённый вопрос — от него зависит вся схема.
2. **Свободные адреса ОЗУ ПП**, не затирающие терминал и систему.
   (Терминал — на ПП! Нельзя его убить, иначе нет вывода.)
3. **Сосуществование** нашего ПП-движка с системным кодом ПП (терминал,
   клавиатура). Прерывания К1 не должны ломать К0.
4. **Опытные примеры**: на форуме zx-pk.ru есть ветки «Портирование Asteroids
   на УКНЦ» (nzeemin), «Last Mission на УКНЦ» (S_V_B) — там наверняка решён
   запуск кода на ПП. Доступ к постам — через регистрацию (у пользователя).

## Следующий шаг (минимальный тест)

Самый маленький проверяемый кусок: **ЦП пишет байт в план → ПП читает его и
рисует один цветной октет.** Если это заработает — вся схема жизнеспособна.
Но сначала надо решить вопрос №1 (как запустить код на ПП).
