# CLAUDE.md — правила работы над проектом MiniCCompiler

Компилятор Mini-C → Macro-11 для УКНЦ (Электроника МС-0511, PDP-11).
Написан на C# (WinForms). Этот файл — инструкция для Claude в будущих сессиях.

---

## ГЛАВНЫЕ ПРИНЦИПЫ (соблюдать всегда)

### 1. Думать до кода
Явно формулировать допущения, указывать альтернативные интерпретации и не
начинать работу, если есть сомнения. При размытом запросе («улучши этот
handler») — спросить, что именно улучшать (производительность, читаемость,
логирование), а не сразу вносить изменения.

### 2. Простота вперёд
Писать минимально возможный код, решающий задачу, без лишних абстракций,
«гибкости» или обработки невозможных сценариев. Если 200 строк можно
сократить до 50 — переписать. Никаких решений «на всякий случай», если они
не запрошены.

### 3. Хирургические правки
Изменять только то, что нужно, и удалять только свой мусор. Не рефакторить
соседние модули «заодно», если это не связано с задачей. Избегать «drive-by»
правок, меняющих код без явной необходимости.

### 4. Не додумывать допущения
Не принимать неверные предположения молча. Если задача неоднозначна —
уточнить у пользователя, а не гадать. Не выбирать интерпретацию и идти
вперёд без подтверждения, не скрывать противоречия.

## КАК МЫ РАБОТАЕМ

- **Рабочий цикл:** Claude правит файлы → кладёт в outputs → пользователь
  скачивает, заменяет в `D:\UKNC\CompMacro11\`, жмёт **Ctrl+Shift+B** в Visual
  Studio, затем **F5** (компиляция Mini-C) и «▶ Эмулятор» (запуск в UKNCBTL).
- **Каждый .c прогоняется через настоящий компилятор** в контейнере перед
  выдачей (см. раздел «Инструмент»). Не только баланс скобок — реальная сборка.
- **Скриншоты — только когда картинка = вопрос** (артефакты, цвета, мусор на
  экране). Результаты тестов, ошибки, «работает/не работает» — текстом.
  Скриншот стоит ~1000+ токенов, текст — десяток.
- **Если у пользователя ошибка, а у Claude компилируется** — это рассинхрон
  файлов (не заменил / не пересобрал). Проверять Output в VS на «Сборка успешна».
- **Тон:** на русском, кратко, по делу, признавать ошибки честно.
- **Папка `Samples`** — пользователь наполняет сам отлаженными проектами.
  Claude туда ничего не добавляет без явной просьбы.

## ИНСТРУМЕНТ: компилятор в контейнере

Контейнер сбрасывается между сессиями — пересоздавать заново:
1. `apt-get install -y dotnet-sdk-8.0` (если битый репозиторий node — удалить
   `/etc/apt/sources.list.d/nodesource.list`, затем `apt-get update`).
2. Скопировать из outputs: Lexer, Parser, AST, CodeGen, CodeGen.Runtime,
   Peephole.cs в `clitest/`. Stdlib.cs не нужен для CLI.
3. csproj net8.0 + `NuGet.config` с `<clear/>` (NuGet заблокирован 403).
4. Main.cs: читает .c, печатает «OK: N chars» или «COMPILE ERROR: ...».
5. `dotnet clitest/bin/Debug/net8.0/minic.dll file.c out.mac`

Mono НЕ собирает C#7 (pattern matching) — только dotnet.
Для отладки рантайма есть эмулятор PDP-11 на Python (load .mac, выполняет
инструкции, детектит зацикливание).

## ЯЗЫК Mini-C — что поддерживается

Типы: `int`, `void`, `bool` (true/false).
Массивы 1D/2D с авторазмером, hex и отрицательными константами.
Управление: if/else, while, for, do-while, switch/case, break, continue, return.
Арифметика: `+ - * / %`. Сравнения. Логика `&& || !` (короткое замыкание).
Биты: `& | ^ ~ << >>`. Присваивания: `= += -= *= /= %= &= |= ^= <<= >>=`.
Инкремент: `++ --`. Тернарный: `cond ? a : b` (вложенный работает).
Функции с рекурсией. `int x, y;` (глоб. и лок.).

**НЕТ:** goto, char[]-строки, запятая в for (`i++,j--`), указатели, struct,
русские символы в строках.

### Встроенные функции
Текст: print_int, print_char, print_str, print_nl, printf (%d %c %s \n),
gotoxy(col,row), setTextColor(c).
Графика: cls, point, line, rect, fill_rect, fill_dither, fill_gradient,
circle, box, init.
Спрайты: sprite(x,y,w,h,ptr) — 5 аргументов; spriteOr(...). Флипов НЕТ.
Ввод: getkey (неблок.), waitkey (ждёт). Пауза: pause.
Математика: abs, min, max, clamp, sin256, cos256, random.
Время: vsync (кадр 50 Гц), getTimer (счётчик кадров).

## КЛЮЧЕВЫЕ ФАКТЫ УКНЦ

- **Окт/дес:** число БЕЗ точки = восьмеричное, С точкой = десятичное.
  `#33` = 27 десятичное, `#33.` = 33. Это источник многих багов!
- **Управляющие коды (десятичные):** ESC=27, CR=13, LF=10, FF=12 (очистка!),
  'Y'=89, '0'=48, пробел=32. Перевод строки = CR(13)+LF(10).
- **Вектор кадрового прерывания = 100** (50 Гц). Регистра LTC @#177546 на ЦП
  УКНЦ НЕТ — обращение к нему = Trap to 4! vsync/getTimer работают через
  перехват вектора 100 с цепочкой к монитору.
- **Экран** 320×264 (режим 0/2), строка VRAM = 80 слов.
- **Видеопорты:** @#176640 адрес, @#176642 данные. Консоль @#177564 статус,
  @#177566 данные. Бит0 = ЛЕВЫЙ пиксель.
- **Цвета текста setTextColor 0..7:** 0=чёрный 1=синий 2=красный 3=magenta
  4=зелёный 5=cyan 6=жёлтый 7=белый. RGB-биты: 1=синий 2=красный 4=зелёный.
- **Цвета графики (cls/point/line 0..3)** — зависят от аппаратной палитры,
  ОТЛИЧАЮТСЯ от текстовых.
- **Знаковое деление:** DIV требует SXT для расширения знака:
  `MOV R0,-(SP) / TST R1 / SXT R0 / DIV (SP)+,R0` (для % добавить MOV R1,R0).
  ASR-деление степеней 2 для отрицательных требует коррекции +n-1 (C-семантика
  «к нулю» vs ASR floor).
- **Формат спрайта:** ширина в единицах по 8px (PixelWidth = Words*8),
  2 бита/пиксель (план0 = младший байт слова, план1 = старший). Подключение
  в код комментарием `// sprite: имя` — редактор экспортирует массив сам.
- **Текст (через 177566) и графику (VRAM) НЕЛЬЗЯ мешать** в одной программе.

## АРХИТЕКТУРА ПРОЕКТА

Файлы C#: Lexer, Parser, AST, CodeGen (+ CodeGen.Runtime — partial-класс со
всем ассемблером рантайма), Peephole, Stdlib, Form1 (IDE), Spriteeditor,
Project, ProjectDialogs, Program.

- CodeGen разбит на partial: `CodeGen.cs` (логика) + `CodeGen.Runtime.cs`
  (метод EmitRuntime). Оба `public partial class CodeGen`.
- csproj — старый формат (файлы перечислены явно).
- **Спрайты:** редактор сохраняет в путь, синхронизированный с тем, откуда
  читает компилятор (проектный `sprites/sprites.spr` или автосейв). При
  создании редактора вызывается UpdateSpritesPath(). Подключение — `// sprite:`.
