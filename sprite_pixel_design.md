# Попиксельный вывод спрайта (некратный x) — дизайн для реализации

Цель: убрать чёрную окантовку у спрайтов с `x` не кратным 8.
Подход — как в `fill_rect`: неполные октеты рисуются **по пикселям** через
выверенный механизм точки (`RTPPNT`). Здесь весь некратный спрайт выводится
попиксельно. Кратный путь (x%8==0) **не трогать** — он идеален.

---

## ГДЕ ВСТРОИТЬ

В `CodeGen.Runtime.cs`, функция RTSPR. Развилка уже есть:

```
        MOV     R0, R1
        BIC     #177770, R1     ; R1 = s = x & 7
        BNE     RTSPS           ; s != 0 -> некратный путь
        ; ... быстрый путь (НЕ ТРОГАТЬ) ...
        JMP     RTSPX
RTSPS:                          ; <-- ВЕСЬ блок от RTSPS до RTSPX заменить
        ; ... старый буферный путь (SPBUF, ROLB) — выбросить целиком ...
RTSPX:
        MOV     (SP)+, R4       ; эпилог (НЕ ТРОГАТЬ)
        ...
```

Заменяешь содержимое между меткой `RTSPS:` и меткой `RTSPX:` на попиксельный
вывод (ниже). Эпилог RTSPX и быстрый путь оставляешь как есть.

---

## ФАКТЫ (выверены по рабочему коду и исходникам эмулятора UKNCBTL)

### Аргументы RTSPR (на входе, через FP=R5):
```
4.(R5)  = x      (пиксели)
6.(R5)  = y      (пиксели)
8.(R5)  = w      (ширина в ПИКСЕЛЯХ, кратна 8)
10.(R5) = h      (строк)
12.(R5) = ptr    (адрес данных спрайта)
```
FP (R5) на входе в RTSPS указывает на кадр — аргументы доступны всю дорогу,
ПОКА не портишь R5. RTPPNT внутри сохраняет R5, так что после вызова R5 цел.

### Формат данных спрайта:
- Массив слов. Слово = `план0 | (план1 << 8)`  (план0 = младший байт).
- Пиксель `col` строки `row`: слово с индексом `row*(w/8) + col/8`.
- Бит пикселя в слове = `col & 7`. Бит 0 = ЛЕВЫЙ пиксель.
- Цвет пикселя (0..3) = `((план0 >> bit) & 1) | (((план1 >> bit) & 1) << 1)`.

### RTPPNT — point(x, y, color). ВЫВЕРЕН, работает на железе.
Вызов (caller-cleans, push справа налево):
```
        MOV     <color>, -(SP)
        MOV     <y>, -(SP)
        MOV     <x>, -(SP)
        JSR     PC, RTPPNT
        ADD     #6, SP
```
RTPPNT **СОХРАНЯЕТ**: R5, R0, R1, R2, R3 (пушит их в прологе).
RTPPNT **НЕ СОХРАНЯЕТ**: R4.
=> Состояние цикла держи в R0..R3 (целы после RTPPNT). R4 (ptr) — сохраняй
сам вокруг вызова, либо пересчитывай адрес слова от ptr каждый раз.

## АЛГОРИТМ (псевдокод, проверен на модели — 0 ошибок)

```
ptr  = 12.(R5)
for row = 0 .. h-1:
    for col = 0 .. w-1:
        word = ptr[ row*(w/8) + col/8 ]      // слово спрайта для пикселя
        bit  = col & 7
        p0   = (word >> bit) & 1             // план0 бит
        p1   = ((word >> 8) >> bit) & 1      // план1 бит (старший байт)
        color = p0 | (p1 << 1)               // 0..3
        RTPPNT(x + col, y + row, color)
```

Замечание: `w/8` = число слов в строке спрайта. Можно вычислить один раз
(`words = w >> 3`) и хранить.

---

## РОСПИСЬ РЕГИСТРОВ (предложение — держать состояние, не стек)

Поскольку RTPPNT сохраняет R0-R3, R5 — держи в них счётчики:
```
R5 = FP (аргументы)         — цел после RTPPNT
R3 = row                    — цел после RTPPNT
R2 = col                    — цел после RTPPNT
R0,R1 = временные (цвет, вычисления) — цел после RTPPNT, но между вызовами
        можешь использовать свободно
R4 = НЕ держать важное через RTPPNT (не сохраняется). Адрес слова спрайта
     лучше пересчитывать от ptr=12.(R5) перед каждым пикселем, либо
     сохранять/восстанавливать R4 вокруг JSR.
```

Адрес слова спрайта (без хранения R4 через вызов):
```
word_index = row * words + (col >> 3)      // words = w>>3
word_addr  = ptr + word_index*2            // ptr = 12.(R5)
word       = @word_addr
```
Умножение `row*words` на PDP-11 без MUL: либо сложением в аккумулятор при
переходе на новую строку (держать «начало строки» = ptr + row*words*2 и
прибавлять words*2 при row++), либо цикл сложения.

РЕКОМЕНДАЦИЯ для простоты: держи `rowptr` = адрес первого слова текущей
строки спрайта. В начале: rowptr = ptr. При row++ : rowptr += words*2.
Тогда адрес слова пикселя = rowptr + (col>>3)*2. Но rowptr надо беречь
от RTPPNT (он в R4 не годится). Вариант: rowptr в памяти (локальная ячейка
на стеке кадра) — читать перед вычислением, не держать в волатильном регистре.

## ИЗВЛЕЧЕНИЕ ЦВЕТА (ассемблерная логика, проверена)

Вход: слово в Rw, bit в Rb. Выход: цвет в Rc.
```
        ; план0 = (слово >> bit) & 1
        MOV     Rw, Rc
        MOV     Rb, Rt          ; Rt = bit (копия, цикл портит)
        BEQ     2$              ; bit==0 -> без сдвига
1$:     ASR     Rc
        DEC     Rt
        BNE     1$
2$:     BIC     #177776, Rc     ; Rc = план0 бит (0/1)
        ; план1 = (старший байт >> bit) & 1
        MOV     Rw, Rt2
        SWAB    Rt2             ; старший байт -> младший
        BIC     #177400, Rt2    ; только план1-байт
        MOV     Rb, Rt          ; bit заново
        BEQ     4$
3$:     ASR     Rt2
        DEC     Rt
        BNE     3$
4$:     BIC     #177776, Rt2    ; план1 бит (0/1)
        ASL     Rt2             ; << 1
        BIS     Rt2, Rc         ; Rc = цвет 0..3
```
Подбери регистры Rw,Rb,Rc,Rt,Rt2 так, чтобы не конфликтовали с row/col.
ASR на PDP-11 — арифметический сдвиг вправо; для бита 0..7 в пределах байта
после BIC старших бит он эквивалентен логическому (знак не вылезет, т.к.
старшие биты уже обнулены BIC перед сдвигами плана1; для плана0 после сдвига
делаем BIC #177776 — лишние биты срезаются). Если осторожничать — можно
заменить ASR на цикл с BIC, но проще проверить на железе.

## КОНТРОЛЬНЫЕ ТОЧКИ ПРИ ОТЛАДКЕ

Тестовый спрайт (полосатый, чтобы видеть смещения):
```c
int STRIPE[16] = {255,255, -256,-256, 255,255, -256,-256,
                  255,255, -256,-256, 255,255, -256,-256};
int SOLID[16]  = {255,255,255,255,255,255,255,255,
                  255,255,255,255,255,255,255,255};
```
- 255   = 0x00FF = цвет1 (план0)
- -256  = 0xFF00 = цвет2 (план1)

Проверь:
1. `sprite(16, y, 16, 8, STRIPE)` — x кратно 8, эталон (быстрый путь).
2. `sprite(20, y, 16, 8, STRIPE)` — s=4, попиксельный. Полосы должны
   совпасть с эталоном, но сдвинуты на 4 пикселя вправо.
3. `sprite(20, y, 16, 8, SOLID)` — окантовки слева/справа быть НЕ должно,
   соседние знакоместа целы.

Ожидание: цвета верные, нет чёрных полос по бокам, нет вертикального
смещения строк.

---

## ПОЧЕМУ ТАК (а не маскирование слов)

Маскирование крайних слов требует точного учёта двухплановой раскладки и
сдвига — много тонких мест, легко ошибиться (проверено: давало мусор/окантовку).
Попиксельный вывод через RTPPNT переиспользует уже работающий механизм точки
(таблицы XWRD, CM0-CM3, CTAB — выверены, на них стоят point/line/fill_rect).
Медленнее (w*h операций вместо w слов на строку), но для спрайтов 16-32px
шириной разница незаметна, а результат гарантированно без окантовки.

Кратный путь (x%8==0) оставлен быстрым (вывод словами) — для него попиксельный
оверхед не нужен.
